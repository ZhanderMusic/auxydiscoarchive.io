Hopefully, the previous/current x-device user % is large enough that you’re still open to doing some QoL for us/them.

 

’Next-Best Thing’…?

Here are a couple (+1) of proposals/requests for your consideration…
(Hopefully, these steer clear of iCloud issues that nixxed auto-syncing.)

TL:DR…

  1. Make users aware if new projects and newer versions of local projects are present in the repo, but not locally.
  2. Make it as easy as possible to Import or Update to the current device.
     
  3. Reinstate auto-syncing as opt-in, and slap an ‘alpha’ or ‘beta’ sticker on it?

 

In more detail…

Use case 1: Project update(s) available

Without the auto-sync, one of the major pain-points is not knowing if I’m about to open the latest version, as it’s difficult to keep track of which device I used to most recently work on a project.

  1. Auxy identifies when a newer (more recently modified) version of a local project exists in the repo (notification dot on the project in the project list view?)
  2. Auxy makes it very simple to fetch/import the newer version, offering the options to Overwrite or Add as New. (Long-press to trigger re-Import, or CTA in side-drag menu?)

 
Use case 2: New projects available

Another major pain-point for multi+device users… picking up projects started on another device.

  1. Auxy identifies projects in the repo that aren’t present locally - e.g. ‘6 other projects found’
  2. Users have the option to view just those non-local projects in a list. (Ideally using the project titles, rather than the file name itself, and would also show the last modified date as reassurance/reminder)
  3. Users can import directly from that list

 
If it’s possible to do all that without stepping out of Auxy’s UI, that’s preferred. (Alternative might be that a ‘Import project’ CTA opens the iCloud repo directory with the Search field pre-filled, to present users with a pre-filtered results list – or exact match.)

Auxy April 6, 2021, 2:55pm #639

I investigated this and it works, except when you import a sample, we convert it to FLAC, which erases the loop flag in the WAV header. So our sampler doesn’t know that the sample should be looped, which is the nicest way to manage it. I even messaged a guy who had made a lot of samples and asked if we could include them in the library, which we might. But then we realised it would be better to convert them to wavetables and make them available as a pack somehow, which is what we plan now. The difference is that we upsample the wavetables to avoid aliasing and this sounds much better than the usual sample pitching.
